# Wedding Photo — дизайн-документ

**Дата:** 2026-07-08
**Статус:** утверждён

## Идея

Веб-приложение для гостей свадьбы (~70 человек). На каждом столе — QR-код (один URL для всех), ведущий на сайт. Каждый гость получает виртуальную «плёнку» на 10 кадров: снимает фото камерой телефона, снимки сразу попадают в общую галерею, которая обновляется в реальном времени. После свадьбы молодожёны скачивают все фото архивом.

## Ключевые решения

| Вопрос | Решение |
|---|---|
| Способ добавления фото | Только камера (живые кадры), без загрузки из галереи телефона |
| Идентификация гостей | Имя при первом входе, сохраняется в localStorage; без регистрации |
| Разблокировка галереи | После первого загруженного фото |
| Лимит | 10 фото на гостя, «Переснять» кадр не тратит; проверка на сервере |
| Админ-функции | Удаление фото + скачивание всех фото ZIP-архивом |
| Сроки | Свадьба через 1–2 месяца |
| Бюджет | Только бесплатные тарифы |

## Стек

- **Next.js (App Router)** — фронтенд + API-роуты; деплой на **Vercel** (free).
- **Supabase (free tier)** — Postgres (метаданные), Storage 1 ГБ (фото, приватный бакет), Realtime (живое обновление галереи).
- Все операции записи и чтения Storage идут через API-роуты Next.js с сервисным ключом; ключи Supabase в браузер не попадают. Галерея получает подписанные URL.

### Отклонённые альтернативы

- **Vite SPA + Cloudinary** — много места (25 ГБ), но нет БД для имён/лимитов, небезопасное удаление без сервера, нет realtime.
- **Firebase** — с конца 2024 бесплатный тариф не даёт Storage новым проектам (требуется карта).

## Экраны и поток

### Гость

1. **Приветствие (`/`)** — имена молодожёнов, дата, объяснение механики («плёнка на 10 кадров»). Ввод имени → создаётся запись `guests`, uuid сохраняется в localStorage. При повторном открытии экран пропускается.
2. **Камера (`/` после входа)** — счётчик кадров «07/10» в стиле окошка плёночной камеры. Кнопка спуска открывает нативную камеру (`<input type="file" accept="image/*" capture="environment">` — работает на iOS Safari и Android Chrome без разрешений). После снимка — превью: «Оставить» (сжатие + загрузка) / «Переснять» (кадр не тратится). Когда кадры кончились — «Плёнка отснята» + переход в галерею.
3. **Галерея (`/gallery`)** — сетка 2–3 колонки, превью с ленивой подгрузкой, под фото имя автора и время. Realtime-подписка на INSERT/DELETE. Тап — полноэкранный просмотр (оригинал) с листанием. Доступна после первого загруженного фото.

### Админ

- **`/admin`** — вход по паролю из переменной окружения. Галерея с кнопкой удаления у каждого фото + «Скачать всё (ZIP)» (архив собирается на сервере из оригиналов).

## Данные

```
guests:  id (uuid, PK), name (text), created_at
photos:  id (uuid, PK), guest_id (FK → guests), storage_path (text),
         thumb_path (text), created_at
```

Storage: приватный бакет `photos`; для каждого снимка два файла — оригинал и превью.

## Поток загрузки фото

1. Снимок → сжатие в браузере через canvas: длинная сторона **2560px, JPEG 90%** (~1 МБ) — качество достаточно для печати до A4.
2. `POST /api/photos` с `guest_id` → сервер **атомарно** проверяет `count(photos where guest_id) < 10` (защита от параллельных запросов и очистки localStorage).
3. Сервер пишет в Storage оригинал + превью 400px (~30 КБ), добавляет строку в `photos`, возвращает новый счётчик.
4. EXIF-ориентация нормализуется при сжатии (фото не должны ложиться боком).

### Ошибки и плохая сеть

- Загрузка с ретраями (3 попытки с паузой), фото держится в памяти страницы до подтверждения сервера; статус «отправляется…».
- При полном отказе — сообщение «не удалось отправить, попробуй ещё раз», кадр не списывается.

## Ёмкость бесплатных тарифов

- Максимум 700 фото × ~1 МБ ≈ 700 МБ из 1 ГБ Storage (реально гости используют ~60–70% лимита).
- Трафик (5 ГБ/мес): сетка галереи грузит превью (~20 МБ на полный просмотр вместо ~700), полный размер — только в полноэкранном режиме.
- Supabase free «усыпляет» проект после ~недели простоя — разбудить заходом в дашборд перед свадьбой.

## API

| Метод | Маршрут | Доступ | Действие |
|---|---|---|---|
| POST | `/api/guests` | публичный | создать гостя по имени |
| POST | `/api/photos` | публичный | загрузка фото (проверка лимита) |
| GET | `/api/photos` | публичный | список фото + подписанные URL |
| DELETE | `/api/photos/[id]` | админ-пароль | удалить фото (Storage + БД) |
| GET | `/api/download` | админ-пароль | ZIP всех оригиналов |

## Визуальный стиль

**Ретро-минимализм, палитра свадьбы: белый + бордовый.**

- Фон — тёплый белый ~`#FAF7F2`; акцент — глубокий бордовый ~`#6E1423` (кнопки, счётчик, имена); текст — почти чёрный; тонкие серо-бежевые разделители.
- Типографика: заголовки — серифный шрифт с кириллицей (Playfair Display / Cormorant, Google Fonts), текст — Inter. Имена молодожёнов крупно, с засечками, бордовым.
- Ретро-детали (дозированно): счётчик-окошко с моноширинными «прокручивающимися» цифрами; круглая бордовая кнопка спуска с кольцом; фото в галерее — белые полароидные рамки с лёгкой тенью, подпись моноширинным шрифтом как штамп даты; едва заметное зерно на фоне; анимация «перемотки плёнки» при исчерпании кадров.
- **Без** фильтров/сепии/виньеток на самих фото — снимки остаются натуральными (материал для альбома).
- Mobile-first: проектируем под телефон, десктоп — центрированная колонка. Крупные зоны нажатия.

## Тестирование

- Юнит-тесты серверной логики: лимит 10 (включая гонки), админ-пароль, валидация загрузки (тип, размер).
- Ручная проверка на реальных iOS Safari и Android Chrome: поведение `capture`, обе ориентации, EXIF-поворот.
- Прогон с network throttling: ретраи и статусы загрузки.
- Генеральная репетиция за ~неделю: 3–5 друзей проходят весь путь по QR со своих телефонов.

## Чек-лист запуска (за 1–2 дня)

1. Разбудить проект Supabase (зайти в дашборд).
2. Загрузить 2–3 затравочных фото, чтобы галерея не была пустой.
3. Очистить тестовые данные через админку.
4. Сгенерировать статичный QR на URL сайта (один для всех столов).
5. Убедиться, что админ-пароль записан у обоих.

После свадьбы: скачать ZIP в первые дни; сайт можно оставить открытым для гостей.

## Вне скоупа

Лайки/комментарии, видео, премодерация фото, мультиязычность, несколько событий на одном сайте.
